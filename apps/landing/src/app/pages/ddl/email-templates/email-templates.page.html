Contact-flow email templates — direction variants

Three takes each for the auto-reply (sent to the submitter) and the admin-notification (sent to me, EN by design). Each variant renders inside a sandboxed <iframe srcdoc> so its CSS behaves like a real email — no inheritance from the host page, inline styles, table-based layout for Gmail/Outlook compatibility. Web fonts are declared but most clients will fall back to the system stack; that fallback is what's actually shown.

V0 in each row is the current production output (baseline reference, not a candidate). V1 / V2 are the two directions to pick between. Pick one per template, or mix-and-match (e.g. V1 auto-reply + V2 admin).

Notes

  • Email clients are CSS hell. Outlook on Windows ignores half of modern CSS, Gmail strips <style> blocks in some configurations. Inline styles + table layout is the lowest-common-denominator that survives. Don't measure against the marketing site — aim for "scannable and on-brand," not "pixel-perfect."
  • Web fonts are aspirational. Most clients fall back to the system stack. The fallback is what shows for 80% of recipients — design for that, treat the web-font render as a bonus.
  • Dark mode not staged here. Worth adding via @media (prefers-color-scheme: dark) inside a <style> block once a winner is picked (works in Apple Mail, native iOS, recent Gmail).
  • Future signature use — these mockups also double as a reference if I ever need a manual signature, an HTML log entry, or a marketing-style transactional template. Keep the file even after a winner graduates.
  • Graduating a winner — once picked, the chosen HTML goes into apps/api/src/modules/email-template/infrastructure/templates/contact-auto-reply.template.ts (and the VI variant) and admin-notification.template.ts. bodyText stays as the plain-text alternative for clients that don't render HTML.
