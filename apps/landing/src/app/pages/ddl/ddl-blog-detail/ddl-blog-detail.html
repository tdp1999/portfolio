Annotated prototypes — best for / failure mode / cost. Read this when comparing layout DNA rather than reading the seeded post.

Variant 1 · Editorial banner centered hero · inline TOC · single-column prose
Best for
Deep dives that earn a ceremonial entrance. TOC + dek + meta on one page = anchored reading.
Failure mode
Wastes space on Notes (TOC of 1 heading + dek that duplicates excerpt). Auto-hide branch required.
Cost
Largest header — ~240-300px before prose starts.
Variant 2 · Dan minimal title + date → prose · everything else in footer
Best for
Short essays + notes where chrome competes with thinking. Reads like Dan Abramov's overreacted.
Failure mode
No TOC = hard to scan long deep-dives. No featured image = pages feel monotonous in a feed.
Cost
Hero ≈ 80-100px. Cheapest layout.
Variant 3 · Sticky meta rail 2-col desktop · sticky TOC + meta + share · collapses <1024
Best for
Deep dives on desktop. Sticky TOC + persistent share make long reads navigable.
Failure mode
Below 1024px the rail must collapse — adds responsive cost. Not ideal for Notes (rail looks empty).
Cost
Hero ≈ 120px. Rail ≈ 240px wide on desktop. Highest CSS complexity.

Variant 1 — Editorial banner

Magazine-style entrance. Pick this when posts deserve ceremony (deep dives, essays) AND the readership benefits from a TOC. The auto-hide branch (Notes or < 800 words) prevents the layout from looking apologetic on shorter posts.

Variant 2 — Dan minimal

Pick this when the writing is the point and chrome competes. Best for opinionated essays, TILs, and tutorials where the reader scrolls top-to-bottom without jumping. No TOC = hostile to long deep dives.

Variant 3 — Sticky meta rail

Desktop-first. Pick this when most readers are on wide screens AND deep dives outnumber notes. The rail carries the metadata work the hero would otherwise inflate, and the sticky TOC keeps long reads navigable. Has a non-trivial mobile collapse cost (rail → top accordion).

Page-level rules (all variants)

Markdown rendering goes through MarkdownService.render() (Shiki highlighter + heading slugify). Anchors inside prose use landingProseAnchors. Copy-link uses landingCopyToClipboard. The share row is a single shared component (landing-ddl-blog-share-row) reused across all three variants — never duplicate the X/LinkedIn markup.

JSON-LD

The BlogPostDetail.jsonLd payload (task 348) ships from the API and is rendered into <script type="application/ld+json"> in the production layout. The <details> blocks below each variant on this DDL prove the payload is plumbed; the script tag injection is a production-side concern not duplicated in this sandbox.
