landing typography · three voices, three jobs

Newsreader · Inter · JetBrains Mono

One serif (Newsreader) for editorial moments, one sans (Inter) for everything readable, one mono (JetBrains) for labels and metadata. A single text node should always have one obvious owner — never reach for "looks nice", reach for the role.

@switch (typographyTab()) { @case ('showcase') {

Use for: hero h1, page titles, pull-quotes.

Don't: buttons, body, repeated card titles.

inter · body / ui

Don't: tags, captions, hero phrase.

jetbrains mono · meta

Aa

pairing rules

  1. One display moment per view. A page gets one Newsreader headline. Sub-headings drop to Inter unless they're inside an editorial reading column.
  2. Eyebrow above display. Mono caps eyebrow + Newsreader headline is the canonical pattern (hero, section openers, project detail header).
  3. Chips never under headings of the same type. A chip cluster (mono caps) belongs in card meta or under a project title — not directly under another mono label.
  4. Pull-quote interrupts body. When using landing-pull-quote inside reading copy, leave 24px padding above/below; the indigo border carries the "this is a moment" cue.
  5. Captions are always mono. Any text labeling a media element (image, code block, diagram) is Mono caps — not Inter italic, not Newsreader.

quick decision table

Question Use
Is this a page-defining headline or hero phrase? Newsreader
Is this a paragraph the reader is meant to read fully? Inter
Is this a button, input, link, or card title? Inter
Is this labeling, tagging, or pointing at something? Mono
Is this a short ALL-CAPS metadata string? Mono
Is this code or a file path? Mono
Is this a quote inside reading copy? Newsreader italic

landing-button for any action that triggers something (submit, toggle, navigate-after-confirm). Reach for solid only when the action is the page's single primary moment — every other CTA is ghost.

landing-link for plain navigation — internal routes, external resources, inline prose. Never style it like a button. Color-only hover, never adds underline. External links automatically get the up-right arrow + target=_blank.

landing-icon-arrow is composed inside button / link, not placed alone. The two glyphs (right, up-right) carry semantic weight: same-domain vs leaving the site. Don't mix.

Indigo budget. A page gets one indigo moment. The arrow's lift-off animation, the active link state, and the solid button background all draw from the same budget — pick one.

} }

Three small label surfaces composed everywhere a piece of metadata appears. Mono caps; one indigo accent rule still applies — none of these claim it.

Hover and click each row. Subtle by design — labels aren't buttons, they're metadata that occasionally accepts a press. Once a winner is chosen these graduate into landing-chip / landing-status-dot; reduced-motion users always get the static state.

Chip

A · Border lift

Border + label brighten one tier on hover. Press settles 1px down. Most restrained option — preserves the "no fill" rule strictly.

B · Tonal fill — shipped

Background fades from transparent to ink-2 on hover. Now live in landing-chip. No cursor:pointer, no click effect — chips are metadata, not buttons; don't lie about clickability.

ANGULAR TYPESCRIPT NESTJS

C · Letter shift

Tracking widens from 0.06em to 0.1em on hover. Pure typographic micro-interaction — feels mechanical, deliberate; only really works on mono.

Status dot

A · Dot pulse (always-on)

On the available state only, the dot pulses gently in a 2.4s loop — a subtle "live signal". No interaction needed; passive presence. Pauses on `prefers-reduced-motion`.

AVAILABLE FOR WORK

B · Radar ring (on hover) — shipped

Hover triggers a single concentric ring expanding outward from the dot, fading as it goes. One-shot, not a loop — feels like a ping rather than a heartbeat. Now live in landing-status-dot on available only.

AVAILABLE FOR WORK

C · Pill brighten

Pill border + label both shift one tier brighter on hover. Mirrors chip option A — the most consistent option if chip A is also the winner. No motion at all, ideal for reduced-motion contexts.

AVAILABLE FOR WORK

recommendation

Chip A · Border lift + Status dot B · Radar ring on available — they share a hover language (one-tier brighten), and the radar adds just enough life to the bio card without a permanent loop. Falls back gracefully under reduced-motion: brighten still applies, ring is suppressed.

} @case ('usage') {

landing-chip for tech tags, categories, topics — anything you'd describe as a "label" rather than a "state". Always mono caps, hairline border, no fill. Cluster them with gap-2; let them wrap.

landing-eyebrow sits ABOVE a heading to flag what the section is about (Selected work, About, Now reading). Pass an array to chain related parts with · separators. Never used as a standalone heading — it's always paired with a Newsreader display below.

landing-status-dot for live operational state — am I taking work, am I in a sprint, am I off the grid. The only pill-radius surface in the design (E4 C2 rule); reach for it sparingly. Use ariaLabel to provide the full sentence for screen readers (the visible label is necessarily terse).

Don't mix. A status indicator is a status-dot, never a chip. A tech tag is a chip, never a status-dot. A section label is an eyebrow, never a chip. The grammar holds the design together.

} }

what is a section rule?

A section rule is the 1px horizontal hairline that separates major content blocks down a long page — hero → about → selected work → contact. Without it, sections bleed into each other; with it, the page reads like a deliberate, paced document instead of a stack.

The plain variant is a quiet hairline that just says "section change". The lift variant adds a 1px indigo stripe on top of the rule AND lifts the section that follows from ink-0 to ink-1 — a tonal step-up that says "what comes next is featured" (used between intro → Selected Work, and copy → Get In Touch, per B2.c).

Mental model: plain = paragraph break · lift = chapter break.

landing-figure for any image with non-trivial context — keep alt meaningful, prefer numbered captions in long-form project detail (FIG. 0X) so prose can refer back. Default to loading="lazy"; only set [preload]="true" for the single hero shot above the fold.

landing-pull-quote interrupts a reading column with one warm sentence in Newsreader italic. The 2px indigo left border is the accent moment for that block — don't pair with a solid button on the same screen. Cite is optional; when used it stays in mono caps, never serif.

landing-section-rule separates major page sections. Use plain by default; reach for lift only when the section that follows deserves to read as a chapter break (Selected Work, Get In Touch). One lift per page is usually plenty.

} }

landing segmented · E5 task 280b

Segmented control

The control DDL uses to switch between Showcase / Prototypes / Usage. Apple-classic by default — recessed track, raised active segment via tone, no shadow. Three visual variants are available; pick per-context.

@switch (segmentedTab()) { @case ('showcase') {

default · variant=apple · 3 segments

2 segments · the no-prototype shape (Showcase / Usage)

with disabled segment · the middle option is unavailable

} @case ('prototypes') {

A · Apple-classic (default)

Recessed ink-1 track inside a hairline border, raised ink-2 active segment. The "raised" feel comes from tone alone — no box-shadow, in line with the project rule.

B · Hairline rectangle

Outlined group with vertical separators between segments. Active fills with ink-2. Less iOS, more technical-spec; pairs well with mono-heavy reading surfaces.

C · Indigo underline

Flat row sitting on a single hairline. The active segment is marked by a 2px indigo underline — closer to editorial tabs. Spends the indigo budget; use sparingly.

recommendation

A · Apple-classic is the default — neutral on indigo budget, works inside cards, reads as a control without claiming editorial weight. Reach for C · Underline only on a dedicated docs page where the indigo moment makes sense; B · Hairline for inline filters in tables / dense data views.

} @case ('usage') {

landing-segmented for switching between mutually exclusive views of the same conceptual unit (Day / Week / Month, Showcase / Prototypes / Usage, Software / Hardware / Services). Don't use it as navigation between unrelated pages — that's a job for landing-link or a regular nav.

Cap segments at 4. Five or more reads as a list; switch to a dropdown / select instead. Disabled segments are supported via the disabled flag on the segment object.

Use mono caps labels. Short, declarative — one or two words. The visual rhythm matches the rest of the landing label vocabulary (eyebrow, chip, status-dot).

Two-way bind active state to a signal. Pattern: [active]="tab()" (activeChange)="tab.set($event)". On a multi-family page like DDL, every family owns its own signal so switching one tab doesn't move the others.

Keyboard. Arrow keys move between segments; Home / End jump to the first / last enabled segment; disabled segments are skipped. Roving tabindex follows the ARIA tablist pattern.

} }
