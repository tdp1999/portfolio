Feed — pagination.

Pagination strategies for any feed/index page (/projects, /blog, future). Three strategies on the same 8-item dataset (page size = 3). All are wired — click around to feel the rhythm. Marks: 🟢 recommend, 💡 fallback, ⚠ do not ship.

V1 · render-all (no pagination) 🟢 recommend for now · <12 projects = render-all wins. No extra UI, scroll is the affordance, SSR prerenders the whole list, footer is always reachable. Add load-more when count crosses ~12.
    @for (project of all; track project.slug) {
  • {{ project.title }} {{ project.startDate.slice(0, 4) }}
    • }
V2 · load-more 💡 fallback (ship when count > 12) · single button at the bottom. Preserves footer access, back-from-detail keeps position, SSR ships first chunk. Avoid: full pagination is overkill, infinite-scroll buries footer.
    @for (project of loadMoreVisible(); track project.slug) {
  • {{ project.title }} {{ project.startDate.slice(0, 4) }}
    • }
V3 · paged (Page 1 of N) ⚠ do not ship · overkill for portfolio scale. Designed for catalogs with 100+ items where users need to jump pages. Forces user to commit to a chunk size, breaks scroll position on each click, SEO needs canonical URLs for each page.
    @for (project of pagedVisible(); track project.slug) {
  • {{ project.title }} {{ project.startDate.slice(0, 4) }}
    • }

Component: <landing-pagination> · auto-ellipsis when total > window (default 7). Tested below with the 8-item dataset — 3 pages × size 3.

Now (≤ 12 projects)
V1 · render-all. Zero ceremony, SSR-friendly, scroll is the only affordance needed.
When count exceeds 12
V2 · load-more. First 12 prerender, "Load more" reveals next chunk. Chunk size = 12.
Never
V3 · paged, infinite-scroll — wrong tools for portfolio. Per session rule "infinite scroll: do not ship".
