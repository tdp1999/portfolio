Four directions for the Cmd+K (Ctrl+K on Win/Linux) search experience. Each variant uses the same mock dataset (pages · sections · projects · actions) and the same keyboard model (↑↓ navigate, ↵ activate, Esc close). Differences are visual chrome, density, and how the kind category is surfaced. Press ⌘K / Ctrl+K anywhere on this page — it opens variant 1.

The chosen direction is Hybrid — a static manifest ships in the bundle (Pages, Sections, Actions) so search feels instant from first keystroke; the BE /api/search endpoint adds Projects + Blog later when content exists. Below shows what each category looks like and where it comes from.

  • pages Build-time manifest. Home, About, Projects, Blog, Uses, Colophon, DDL. Always present. ~7 entries · ~1 KB.
  • sections Each page declares its anchor sections (eg. /#stack, /#get-in-touch) in a shared registry. ~10–20 entries.
  • actions Client-side definitions: toggle theme, switch language, copy email, download CV, open GitHub, scroll to top. Execute on activation. ~5–10 entries.
  • projects Lazy — debounced fetch to /api/search?q=…&kind=project on input. Returns title, slug, year, stack chips.
  • blog Same shape as projects — lazy API. Empty until first post ships.

Filter algorithm: case-insensitive substring on title + description + kind. Production should upgrade to a fuzzy matcher (Fuse.js / minisearch / a custom trigram index) — gives typo tolerance and substring scoring.

Centered modal, top-third position. Section headers above each group ("Pages", "Projects", "Actions"). Subtle hover + selected states. Keyboard hint footer (↑↓ ↵ esc). Standard pattern most users recognize.

v1 · linear-style

Top-third position like macOS Spotlight. No section headers — flat list with a small uppercase mono "kind" chip on the right of each row. Less chrome, more speed. No footer.

v2 · spotlight (flat)

Wider modal. Each row has a framed icon on the left, two-line content (title + description), and a hint on the right. Sections have subdued mono headers. Best when items have meaningful descriptions to disambiguate.

v3 · raycast (rich rows)

Centered modal, narrower. Just the input and a flat list of titles. No icons, no descriptions, no chips, no footer. Closest to a Linear-clone with all chrome stripped. Quiet, fast, opinionated.

v4 · minimal
Variant Density Chrome Vibe Pick when
v1 · linear medium medium standard · trusted Default. Familiar pattern. Section headers help when items span many categories.
v2 · spotlight high low fast · OS-native Power-user vibe. Few items per query. Kind chip on the right disambiguates without group headers.
v3 · raycast low (tall rows) high rich · expressive Descriptions add meaningful context. Best when items overlap by name (multiple "Stack" entries, etc.).
v4 · minimal highest zero editorial · quiet You know your data well; you want the palette to disappear visually and just deliver results.

Leaning: v1 (Linear-style) as the likely canonical pick — it pays off the category richness of the underlying data (pages, sections, actions, projects) without committing to tall rows. v2 (Spotlight) is the strong alternative if you want the palette to feel like an OS tool. Still exploring; no commitment yet.
